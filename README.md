### Отчёт: арифметика для MNT4-753 (Fp, Fp2, Fp4) в Solidity

#### Общее
Выполнена реализацию базовой и расширенной арифметики для кривой MNT4-753. 

Сборка/компиляция и тестирование выполнены в Foundry со следующими настройками:
- `solc = 0.8.33`
- `optimizer = true`, `optimizer_runs = 200`
- `via_ir = true`
---

### Структура реализованных компонентов

#### 1) `src/BigIntMNTV2.sol` — базовое поле Fp (Montgomery arithmetic, 3×256-bit limbs)

##### Назначение
Реализует арифметику по модулю простого числа `p` для MNT4-753. Элемент поля представляется как 3 “лимба” по 256 бит (little-endian):
- число `x` кодируется как `x = x0 + x1·2^256 + x2·2^512`,
- модуль `p` имеет длину 753 бита.

Все тяжёлые операции (умножение/квадрат) реализованы в Montgomery domain, что позволяет выполнять:
- `montMul(a, b) = a·b·R^{-1} mod p` при фиксированном `R = 2^(256·3) = 2^768`.

##### Montgomery-параметры
В контракте заданы константы:
- `P_0, P_1, P_2` — модуль `p` по лимбам.
- `R2_0, R2_1, R2_2` — `R^2 mod p` (для перевода в Montgomery-область).
- `MAGIC` — `-p^{-1} mod 2^256` (для шага REDC).

Корректность параметров проверялась генератором констант на Python и сравнением с кодом.

##### Инварианты (предусловия)
Для скорости операции сложения/вычитания реализованы в предположении, что входы уже редуцированы:
- для `add/sub`: предполагается `0 <= a < p` и `0 <= b < p`.
- выходы также редуцированы в диапазон `[0, p)`.

##### API (функции и назначение)

###### Низкоуровневые (stack-friendly) функции
- `add3(a0,a1,a2, b0,b1,b2) -> (r0,r1,r2)`  
  Сложение в Fp: `(a + b) mod p`. Выполняет 3-лимбовое сложение с переносом и условную редукцию (одна проверка/одно вычитание `p`).

- `sub3(a0,a1,a2, b0,b1,b2) -> (r0,r1,r2)`  
  Вычитание в Fp: `(a - b) mod p`. Выполняет 3-лимбовое вычитание с заёмом и при отрицательном результате добавляет `p`.

- `montMul3(a0,a1,a2, b0,b1,b2) -> (r0,r1,r2)`  
  Montgomery multiplication: `a·b·R^{-1} mod p`.

- `montSqr3(a0,a1,a2) -> (r0,r1,r2)`  
  Montgomery squaring: `a^2·R^{-1} mod p`.  

###### Высокоуровневые обёртки (на `uint256[3]`)
- `add(uint256[3] a, uint256[3] b) -> uint256[3]`
- `sub(uint256[3] a, uint256[3] b) -> uint256[3]`
- `montMul(uint256[3] a, uint256[3] b) -> uint256[3]`
- `montSqr(uint256[3] a) -> uint256[3]`
- `toMontgomery(uint256[3] x) -> uint256[3]`  
  Перевод в Montgomery: `x * R^2 * R^{-1} = x·R mod p`.

- `fromMontgomery(uint256[3] x) -> uint256[3]`  
  Выход из Montgomery: `x * 1 * R^{-1} = x·R^{-1} mod p`.

##### Основные оптимизации в `BigIntMNTV2`
1. **Оптимизированный `mul512` для EVM**  
   Вместо разбиений на 128-битные части используется приём:
   - `lo = mul(a,b)`
   - `hi = mulmod(a,b, 2^256-1) - lo - borrow`
   Это уменьшает gas на “сборку” 512-битного произведения.

2. **Montgomery REDC для фиксированного `n=3`**  
   REDC развернут под 3 итерации (по числу лимбов), без циклов по лимбам в Solidity-уровне.

3. **Специализированный квадрат `montSqr3`**  
   Использует симметрию произведения (кросс-термы удваиваются) и отдельную REDC.  
   Выигрыш по gas небольшой, но квадрат частая операция в Miller loop.

4. **Yul/assembly реализация carry/borrow**  
   Используются `adc/sbb` на `lt`/`add`/`sub`, что дешевле высокоуровневых проверок.

#### 2) `src/MNT4ExtensionV2.sol` — расширения Fp2 и Fp4

##### Назначение
Реализует арифметику в расширениях:

- `Fp2 = Fp[u] / (u^2 - 13)`  
  Элемент `a ∈ Fp2` имеет вид `a = a0 + a1·u`, где `a0, a1 ∈ Fp`.

- `Fp4 = Fp2[v] / (v^2 - u)`  
  Элемент `x ∈ Fp4` имеет вид `x = c0 + c1·v`, где `c0, c1 ∈ Fp2`.

Коэффициенты хранятся в Montgomery-форме над `Fp`, т.е. все базовые операции используют `BigIntMNTV2`.

##### Типы данных
- `struct Fq2 { uint256[3] c0; uint256[3] c1; }`
- `struct Fq4 { Fq2 c0; Fq2 c1; }`

##### API (функции и назначение)

###### Вспомогательные операции над Fp
- `fpMulBy13(x0,x1,x2) -> (y0,y1,y2)`  
  Быстрое умножение на 13 в `Fp` через сложения/удвоения (без `montMul`).

###### Fq2
- `fq2Add(Fq2 a, Fq2 b) -> Fq2`
- `fq2Sub(Fq2 a, Fq2 b) -> Fq2`

- `fq2MulTo(Fq2 out, Fq2 a, Fq2 b)`  
  Умножение в `Fp2` с записью в заранее выделенную структуру `out` (для минимизации аллокаций в больших циклах).

- `fq2Mul(Fq2 a, Fq2 b) -> Fq2`  
  Обёртка над `fq2MulTo`.

- `fq2SqrTo(Fq2 out, Fq2 a)`  
  Квадрат в `Fp2` по формуле:
  - `(a0 + a1·u)^2 = (a0^2 + 13·a1^2) + (2·a0·a1)·u`

- `fq2Sqr(Fq2 a) -> Fq2`  
  Обёртка над `fq2SqrTo`.

- `fq2MulByUTo(Fq2 out, Fq2 x)` и `fq2MulByU(Fq2 x) -> Fq2`  
  Умножение на `u` в `Fp2`:
  - `(x0 + x1·u)·u = (13·x1) + x0·u`

- `fq2MulByFpTo(Fq2 out, Fq2 a, uint256[3] s)` и `fq2MulByFp(Fq2 a, uint256[3] s) -> Fq2`  
  Умножение элемента `Fp2` на скаляр `s ∈ Fp`.

###### Fq4
- `fq4Add(Fq4 a, Fq4 b) -> Fq4`
- `fq4Sub(Fq4 a, Fq4 b) -> Fq4`

- `fq4MulTo(Fq4 out, Fq4 a, Fq4 b)`  
  Умножение в `Fp4` (Karatsuba на уровне `Fp2`), с использованием `out` как scratch:
  - минимизируются временные аллокации,
  - расчёт организован так, чтобы меньше занимать стек (актуально при `via_ir=true`).

- `fq4Mul(Fq4 a, Fq4 b) -> Fq4`  
  Обёртка над `fq4MulTo`.

- `fq4SqrTo(Fq4 out, Fq4 a)`  
  Квадрат в `Fp4` по формуле:
  - `(c0 + c1·v)^2 = (c0^2 + u·c1^2) + (2·c0·c1)·v`, где `v^2 = u`

- `fq4Sqr(Fq4 a) -> Fq4`  
  Обёртка над `fq4SqrTo`.

- `fq4MulByFq2To(Fq4 out, Fq4 a, Fq2 s)` и `fq4MulByFq2(Fq4 a, Fq2 s) -> Fq4`  
  Умножение `Fp4` на “скаляр” `Fp2`:
  - `(a0 + a1·v)·s = (a0·s) + (a1·s)·v`

- `fq4MulByV(Fq4 a) -> Fq4`  
  Умножение на `v` в `Fp4`:
  - `(c0 + c1·v)·v = (u·c1) + c0·v`  

##### Основные оптимизации в `MNT4ExtensionV2`
1. **Karatsuba умножение в Fq2 и Fq4**  
   Уменьшает число дорогих `montMul`:
   - `Fq2 mul`: 3 Montgomery mul вместо 4 (классического).
   - `Fq4 mul`: строится на `Fq2` и также экономит `Fq2Mul`.

2. **Специализированные квадраты `fq2Sqr`, `fq4Sqr`**  
   Квадрат заметно дешевле общего умножения и используется массово в Miller loop.

3. **Out-parameter (`...To`) API и использование результата как scratch**  
   Идея: в больших циклах заранее выделять память и переиспользовать её, избегая постоянных новых `memory`-объектов.

4. **Оптимизация умножения на малую константу и базисные элементы (`*13`, `*u`, `*v`)**  
   Получилось сильное снижение газа (~x7 раз).

### Методология разработки и проверки

#### Генерация и валидация констант
- Python-скрипты использовались для генерации:
  - `R = 2^(256·3)`, `R2 = R^2 mod p`, `MAGIC = -p^{-1} mod 2^256`,
  - проверка `p` на простоту и корректность длины.

#### Тестирование корректности
- Unit-тесты для `Fp`: сложение/вычитание/переходы Montgomery/умножение/квадрат.
- Unit-тесты для `Fp2/Fp4`: тождественность, сравнение `sqr(a)` и `mul(a,a)`, корректность `mulByV`.
- Fuzz-тесты:
  - ассоциативность `Fq2Mul` и `Fq4Mul`,
  - дистрибутивность в `Fq2`.

#### Профилирование по gas
- Использован `forge test --gas-report`.
- Добавлены “loop-like” bench функции, чтобы приблизить измерения к реальным нагрузкам pairing и минимизировать шум от `external` overhead.

##### Интерпретация bench-результатов (амортизированная цена одной операции)
`bench` измеряет множество операций в одном вызове, минимизируя overhead ABI/external call.

- `benchFq4Mul(32)`:
  - `1464024 / 32 ≈ 45,751 gas` за 1 `Fq4Mul` (в цикле)
- `benchFq4Sqr(64)`:
  - `2601889 / 64 ≈ 40,655 gas` за 1 `Fq4Sqr` (в цикле)

Ключевые наблюдения:
- `Fq2Sqr` дешевле `Fq2Mul` (~8–9%).
- `Fq4Sqr` существенно дешевле `Fq4Mul` (~14%).
- `fq4MulByV` примерно в 7 раз дешевле полного `fq4Mul`.
